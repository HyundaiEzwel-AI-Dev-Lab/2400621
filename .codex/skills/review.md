# Review

H-PMS 프로젝트의 코드 변경 또는 특정 영역을 리뷰할 때 이 절차를 따른다.

## Purpose

- 버그, 회귀 위험, 테스트 누락, 보안/권한 문제, 유지보수 리스크를 찾는다.
- 수정 작업이 아니라 리뷰 결과 산출에 집중한다.

## Use Cases

- PR 또는 `git diff`를 리뷰할 때
- 특정 파일이나 기능 구현 결과를 검토할 때
- 릴리스 전 위험 요소를 점검할 때
- 변경 후 테스트 누락이나 회귀 가능성을 확인할 때

## Workflow

### 1. Plan

리뷰 범위를 확인한다.

- 리뷰 대상 diff, 파일, 기능, 브랜치
- 중점 검토 영역
- 실행 가능한 검증 명령
- 제외할 범위

리뷰 계획을 제시한다.

- 확인할 파일 또는 변경 영역
- 결함 판단 기준
- 실행할 검증 명령
- `docs/worklog.md` 기록 방식

### 2. Approval

리뷰 범위와 검증 명령 실행 여부를 사용자에게 확인한다. 리뷰만 요청받은 경우 승인 없이 코드를 수정하지 않는다.

### 3. Execute

코드를 읽고 결함 중심으로 검토한다.

우선순위:

1. 실제 버그 또는 회귀
2. 보안, 권한, 데이터 무결성 문제
3. 테스트 누락
4. 유지보수 리스크

각 finding에는 가능한 한 파일과 라인, 영향, 재현 조건 또는 실패 시나리오를 포함한다.

### 4. Verify

필요한 범위에서 검증 명령을 실행한다.

```bash
git diff --stat
git diff --check
cd frontend && npm test
cd frontend && npm run type-check
cd frontend && npm run build
```

리뷰 특성상 모든 명령을 실행하지 않을 수 있다. 실행한 명령과 생략한 명령의 이유를 보고한다.

### 5. Record

`docs/worklog.md`에 기록한다.

- 날짜
- 리뷰 범위
- 주요 발견사항 요약
- 실행한 검증 명령
- 남은 질문 또는 위험

리뷰 결과는 심각도순 findings를 먼저 제시한다.

## Forbidden

- 요청 없는 코드 수정을 하지 않는다.
- 스타일 취향만으로 finding을 만들지 않는다.
- 파일/라인 또는 명확한 근거 없는 결함을 단정하지 않는다.
- 낮은 중요도의 의견을 실제 결함보다 앞세우지 않는다.
- 테스트 실패 또는 검증 실패를 숨기지 않는다.

## Deliverables

- 리뷰 범위
- 심각도순 findings
- 파일/라인 근거
- 테스트 누락 또는 회귀 위험
- open questions
- 실행한 검증 명령과 결과
- `docs/worklog.md` 기록

## Completion Criteria

- 리뷰 범위를 명확히 했다.
- 결함 중심으로 결과를 제시했다.
- 실행한 검증 결과를 보고했다.
- 요청 없는 코드를 수정하지 않았다.
- `docs/worklog.md`를 업데이트했다.
